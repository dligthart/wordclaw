# Agent Commit Standards

In order to power the automated Semantic Release pipeline safely, WordClaw enforces a strict adherence to **Conventional Commits**. Because autonomous AI agents (like yourself) author the vast majority of commits in this repository, understanding these rules is critical. 

Failure to format a commit message correctly will cause the release pipeline to fail to parse the version bump, halting CI/CD distributions.

## The Anatomy of an Agent Commit

All commits generated by AI assistants acting on the WordClaw ecosystem must follow this structure:

```text
<type>(<optional scope>): <description>

[optional body explaining the "why", not just the "what"]

[optional footer(s) for issue tracking or breaking changes]
```

### 1. Types (The Trigger Words)
The `<type>` defines what level of Semantic Versioning bump will be triggered by the automated release pipeline. You **must** choose from the following explicitly approved types:

#### Major & Minor Triggers (Releases App)
* **`feat`**: A new feature for the user. (*Triggers a MINOR version bump, e.g., 1.1.0 -> 1.2.0*)
* **`fix`**: A bug fix for the user. (*Triggers a PATCH version bump, e.g., 1.1.0 -> 1.1.1*)
* **`perf`**: A code change that mathematically improves performance. (*Triggers a PATCH version bump*)

#### Maintenance Triggers (No Version Bump)
These prefixes cleanly organize the repository history but will intentionally **skip** triggering a SemVer bump, as they do not affect the public API or end-user runtime:
* **`docs`**: Documentation only changes (e.g., updating a `doc/rfc` or `README.md`).
* **`style`**: Changes that do not affect the meaning of the code (white-space, formatting, missing semi-colons, etc).
* **`refactor`**: A code change that neither fixes a bug nor adds a feature.
* **`test`**: Adding missing tests or correcting existing tests.
* **`ci`**: Changes to our CI configuration files and scripts (e.g., GitHub actions, `.releaserc.json`).
* **`chore`**: Other changes that don't modify `src` or test files (e.g., updating dependencies).

### 2. Scopes
The scope provides additional context to the location of the change. While optional, agents are highly encouraged to use them.
Common scopes in the WordClaw architecture include:
* `(api)`: Changes relating to Fastify REST or MCP routes.
* `(ui)`: Changes to the SvelteKit Supervisor frontend.
* `(db)`: Drizzle ORM schemas or PgVector migrations.
* `(auth)`: L402 payments or JWT verifications.
* `(docs)`: Documentation adjustments.

### 3. Descriptions
The description must immediately follow the colon and space. 
* Use the imperative, present tense: "change" not "changed" nor "changes".
* Do not capitalize the first letter.
* Do not place a period `.` at the end.

### 4. Breaking Changes
If an agent makes a change that breaks backward compatibility (e.g., changing a core API response schema or database structure), the commit **MUST** include a breaking change footer. 

Adding this footer automatically triggers a **MAJOR** SemVer bump (e.g., `1.2.0` -> `2.0.0`).

```text
feat(api): overhaul the semantic search response data schema

BREAKING CHANGE: The `/api/search/semantic` endpoint now returns `similarity` instead of `distance`, and `textChunk` instead of `text`. Frontend applications must update their mapping interfaces to prevent NaN errors.
```
*(Alternatively, you can append a `!` to the type/scope indicator: `feat(api)!: overhaul...`)*

## Examples of Perfect Agent Commits

- **Good Patch**: `fix(ui): resolve NaN rendering bug when calculating float similarity scores`
- **Good Feature**: `feat(db): append caching layer for semantic search endpoints`
- **Good Chore**: `chore(deps): bump semantic-release to v24.1.2`
- **Good Docs**: `docs(rfc): draft proposal 0024 for horizontal scaling metrics`

## Summary Checklist for Agents
Before firing a `git commit -m` system command, agents should silently verify:
1. Did I use an exact permitted prefix (`feat`, `fix`, `docs`, `chore`, `refactor`, `test`, `ci`, `perf`)?
2. Did I keep the entire first line completely lower-case?
3. Did I avoid using a period at the end of the first line?
4. If this breaks compatibility, did I explicitly include the `BREAKING CHANGE:` footer?
